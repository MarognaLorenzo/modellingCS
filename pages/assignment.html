MCS25 Project description

About the MCS project in general


The project is on modeling and reasoning on temporal systems using LTC and IDP3. It consists of two parts:

Part one: Simple static system: May be done in teams of 2, or individually. You will be given an informal but precise description of a small domain, along with a simple given vocabulary and visualization tools.

Each group designs a solution consisting of: An extended vocabulary consisting of types, predicate, and/or function symbols, along with informal interpretations written in comments. A simple theory aligned with the given description of the system. Some example structures to test your theory with. This part will be corrected through a combination of manual inspection and automatic testing. A model solution will be provided afterward.

Part two: Complex LTC system: Individual work. You will be given a temporal extension and refinement of the system from the first part. You will be provided with a formal base vocabulary (which you will be allowed to extend). Each student designs an LTC solution and submits it. For this part, you will be provided with test examples, and visualisation of solutions (this should help you design the correct specification). This part will be automatically corrected. A model solution will be provided afterward.

Rules The project is obligatory; you cannot participate in the exam unless you have submitted the 2 parts of the project! Together, the two project parts will take up to 30 hours. Please keep track of your time on the project and warn us if you spend more. There is no way to redo the project in the summer, unless for compelling reasons. There is no way to transmit scores of the project from last year. You are allowed to discuss this project with others. However, you are not allowed to make your code openly available on the web. If you want to use git, do not use public repositories.

Grading The project is graded on 5/20 points:

  • (3,5p) points are for the part during the semester (2 parts of the project).
    • (1.0p) First part - Corrected through a combination of manual inspection and automatic testing. Design choices will be graded (sensible vocabulary, proper choices between function and predicates, etc.)
    • (2.5p) Second part - Automatically corrected. The correctness of the solution will be tested with model checking.
  • (1.5p) points are for questions in the exam, on temporal logic CTL and LTL and/or refinement.

About the concrete MCS project 2025


As a this year (2025) project assignment for the course MCS we will model the card game named "Set". This is a turn-based multiplayer game with a custom deck, and hence suitable to represent as a linear temporal system. Following are the explanations of cards, how they form a "set", and game rules used for this project.

Cards in this game are described by four properties (Number, Color, Shade, Shape). Each of the properties has three possible values: Number is either 1, 2, or 3; Color is either Red, Green, or Orange; Shade is either Empty, Stripes, or Full; Shape is either Diamond, Squiggle, or Oval. These properties describe how many and what kind of objects can be found on the card.

For example, the following card contains One Oval Full Red object(s).

Three cards form a set (of cards) if for each property either:

  • All three have the same value for that property.
  • All three have distinct values for that property.

For example, the following triple is not a set!

For example, the following triple is a set!

For this project, we proceed with the following variation of the game rules:

  • There are n players (n > 1).
  • Initially, the score of each player is zero.
  • Initially there are nine cards at the card table.
  • Players take turns one by one in a circular manner.
  • When on a turn, the player has two options.
    1. Guess if something is a set.
    2. Claim there are no sets at the card table.
  • If the player guessed correctly, their score gets increased by 3 points plus 1 point for each player in a chain immediately before them that claimed there is no set at the table (this resets if cards are added to the table).
  • If a player guessed wrongly, their score gets reduced by 1 point.
  • If all n players consecutively claim that there is no set at the table, the player that is next on the move (the one that claimed first) deals three cards to the table and stays on turn (to guess or claim no set).
  • The game is over when there are no more cards to deal with, and the loop of n players claims there is no set at the table.
  • The winner is the player with the biggest score.
  • The tiebreaker is the last set guessed (who is the last to guess the set, they win).

About the part one


For the first part of the project, your task is to develop basic concepts for the "Set game" using the IDP3 system. The first task is extending the vocabulary (under the button "1. Cards"). Choosing the symbols in the right way is important for this step, and it will be reflected in your grade. As the second step, you will have to model the notion of the "set" of cards. Finally, you will model a set of all "sets" among some group of cards (i.e., cards that are on the table).

The idea is that you can gradually implement functionalities of the actions/buttons in the interface. For example, once you are done with the "2. Sets" theory, the button "Check selected" should work correctly. Similar for the "3. Card table" theory and buttons "Are there sets" and "Show sets".

Opening different parts of your task "1. Cards", "2. Sets", and "3. Card table" will also present a short description of the purpose of that part. Often you will find hints and suggestions along the comments in the IDP code, so make sure to read them.
